## Feasibility Review Messages

*Added 2026-04-22. Decision 023. Triggered when Product bot signs off on Stage 2.*

### Product bot → UX Designer (in #design, simultaneous with Architect message)
```
Requirements are frozen for {feature_name}.

Before you start designing, I need your read on feasibility. Here's what's specified:

{use_case_summary}

Two questions:
1. Anything in here you can't deliver as described?
2. Any red flags I should know about before we proceed?

48 hours to respond — or let me know if you need more time.
```

### Product bot → Architect (in #tech-planning, simultaneous with UX message)
```
Requirements are frozen for {feature_name}.

Before design starts, I need your read on technical feasibility. Here's what's specified:

{use_case_summary}

Two questions:
1. Anything architecturally that can't be delivered as described?
2. Any red flags I should know about before we proceed?

48 hours to respond — or let me know if you need more time.
```

### UX Designer / Architect response: All clear
```
Looks good from my side — no issues with {feature_name} as specified. Ready to proceed.
```

### UX Designer / Architect response: Minor red flag
```
One thing worth flagging on {feature_name} — it's minor and I can handle it:

{plain_english_flag}

My recommendation: {proposed_resolution}

I'll proceed on that basis unless you'd like to discuss.
```

### UX Designer / Architect response: Major red flag
```
I've found something on {feature_name} that needs a decision before I can proceed:

{plain_english_flag}

This affects: {scope_or_timeline_impact}

Options:
1. {option_1}
2. {option_2}

I can't move forward until this is resolved.
```

### Ops → founder: major red flag escalation (Hands-off mode)
```
{reviewer_role} flagged something on {feature_name} before design starts.

The issue: {plain_english_flag}

The team recommends: {recommended_option}

Happy to proceed on that basis — just let me know.
```
CTA: `Yes, go ahead`  ·  `I want to discuss`

### Ops → founder: major red flag escalation (Balanced mode)
```

{plain_english_flag}

Options:
• {option_1}
• {option_2}

The team leans toward option {n} — but it's your call.
```
CTA: `Option 1`  ·  `Option 2`  ·  `Let's discuss`

### Ops → founder: major red flag escalation (Hands-on mode)

## Ops Bot — Greeting Messages

### Startup template onboarding greeting (in #ops):
```
Hey! I'm Riley, your team coordinator.

Your team is ready. Here's who you've hired:

• {name_1} — {role_1} (#{channel_1})
• {name_2} — {role_2} (#{channel_2})
• {name_3} — {role_3} (#{channel_3})

You can head to any channel directly — or just tell me what you need here and I'll make sure it gets to the right person.
```

### Ops bot: post-onboarding team readiness message (fires once, immediately after greeting)
```
One more thing — a few tools will help your team do their best work:

{tool_line_1}
{tool_line_2}
{tool_line_3}

Want to set these up now, or should I remind you when someone needs them?
```
CTA: `Set up now`  ·  `Remind me when needed`

**Tool line format** (one per relevant integration):
- Connected: `✓  Figma — Jordan is all set`
- Not connected: `○  Figma — helps Jordan produce design files (optional)`
- Required, not connected: `○  GitHub — needed for Sam, Morgan, and Casey to take action`

**Rules**:
- Only list integrations relevant to the hired template
- Always list GitHub first if not connected
- Skip any integration that is already connected
- If all optional tools are connected, skip this message entirely
- This message fires once only — never repeated

### Ops bot: runtime tool blocker (first time a bot needs a missing tool)
```
To let {bot_name} do this, they'll need {tool_name} connected.
It takes about a minute.

In the meantime, {bot_name} has shared what they can without it.

[Connect {tool_name}]   Skip (use {degraded_format} instead)
```

### Ops bot: batched tool blocker (multiple tools needed at once)
```
A couple of tools would help your team right now:

○  {tool_1} — {bot_name_1} needs this to {plain_action_1}
○  {tool_2} — {bot_name_2} needs this to {plain_action_2}

Both teammates have shared what they can without them.

[Set up tools]   Skip for now
```

### Ops bot: tool connected confirmation (after founder connects a tool)
```
✓  {tool_name} is connected. {bot_name} is all set — they can continue now.
```

### Ops bot: routing to a single teammate
```
That sounds like one for {bot_name} in #{channel}.

I've let them know. Head to #{channel} to follow along — or stay here and I'll keep you updated.
```

### Ops bot: routing to multiple teammates simultaneously
```
That sounds like one for {bot_name_1} ({role_1}) and {bot_name_2} ({role_2}).

I've let them both know. {bot_name_1} will {plain_action_1} and {bot_name_2} will {plain_action_2}.

Head to #{channel_1} or #{channel_2} to follow along — or stay here and I'll keep you updated.
```

### Ops bot: founder asks something Ops can handle directly (team status)

```
Here's what your team is working on right now:

• {bot_name_1} ({role_1}) — {current_task}
• {bot_name_2} ({role_2}) — {current_task}
```

### Ops bot: founder asks something completely outside any role

```

{relevant_answer_or_next_step}
```

## Per-Bot Wrong-Channel Routing Templates

Used when a founder sends a message to the wrong bot. Every bot uses this pattern — never a hard refusal.

### When message has partial relevance (bot answers what it can + routes the rest):
```
{role_relevant_response}

For the {out_of_scope_part}, {routing_bot_name} in #{routing_channel} is the right person — I've let them know.
```

**Example (Engineering bot receives a wireframe request)**:
```
From my side, the login screen needs to handle magic-link 
callbacks and session persistence — happy to spec those 
technical requirements so Jordan has context to work from.

For the wireframes themselves, Jordan in #design is the 
right person — I've let them know.
```

### When message has zero relevance (full redirect):
```
{routing_bot_name} in #{routing_channel} is much better placed for this than I am — I've let them know you need help with {topic}.

[From my {role} perspective, {brief_relevant_angle}.]
```

**Example (Security bot receives a roadmap question)**:
```
Roadmap decisions are Alex's domain in #product — I've 
let them know.

From a security angle, I'd push for auth hardening before 
you scale — happy to feed that into the planning 
conversation if it helps.
```

### Trailing line all bots use after routing:
```
Not sure who to ask next time? Riley in #ops always knows where to send you.
```

### Engineering bot: first GitHub connection — sensitive area scan (in #engineering)
*Shown immediately after CI/CD check (or after CI setup completes). Only shown if repo contains recognisable sensitive patterns. Skipped entirely if none found.*
```
I've had a look around your codebase.

A few areas usually need extra care. Should I always flag it when I'm about to make changes in any of these?

{area_row_1}
{area_row_2}

```

**Area row format** (one per detected area):
```
[ ] Payments  — billing, subscriptions, pricing
[ ] Auth      — login, sessions, user accounts  
[ ] Admin     — internal tools and dashboards
[ ] Security  — keys, configs, access controls
```
CTA: `Save preferences`  (checkboxes pre-unchecked; founder opts in)

```

```

### Engineering bot: first GitHub connection — no sensitive areas found
*Shown only if scan finds no recognisable sensitive patterns.*
