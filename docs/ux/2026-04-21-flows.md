## Flow 1: First-Time Onboarding (The First 10 Minutes)

**Entry**: Founder clicks link in magic-link email.  
**Exit**: Founder is in their first channel, Ops bot has greeted them.  
**Success metric**: Founder sends their first message within 10 minutes of clicking the email link.

```
[Magic Link Email]
        │
        ▼
[Welcome Screen]
  Enter email → "Send me a sign-in link"
        │
        ▼
[Check Email Screen]
  "Check your inbox — we sent you a link"
        │
        ▼
[Magic Link Click → Supabase Auth Callback]
        │
        ▼
[Step 1: Name your company]
  Text input → Continue
        │
        ▼
[Step 2: Pick a team template]
  Startup  /  Enterprise  /  Blank
  Each card shows team roles + "Works with:" tool line
        │
        ▼
[Step 3: How do you like to work?]
  Hands-off  /  Balanced (recommended, pre-selected)  /  Hands-on
  One tap to select — no form fields
        │
        ▼
[Step 4: Connect GitHub]
        ├── Connect GitHub → [GitHub App Install flow] → return → Continue
        └── Skip for now → Continue
        │
        ▼
[Step 5: Meet your team]
  Ops bot greets founder with team list
  "Go to my workspace"
        │
        ▼
[Main Workspace — #ops channel]
  Ops bot intro message visible
  Sidebar shows all seeded channels
```

**Drop-off guards**:
- GitHub step has a prominent "Skip for now" — never block on GitHub
- Back button on every step
- Progress dots always visible (5 steps, not 8)
- No email confirmation required — magic link IS the confirmation

## Flow 2: Sending a Message to a Bot (Core Loop)

**Entry**: Founder is in a channel, types a message.  
**Exit**: Bot has responded (with or without a plan).

```
[Founder types in message input]
        │
        ▼
[POST /api/bots/message]
        │
        ├── Action cap exceeded?
        │         │
        │         ▼
        │   [Inline warning: "Your team has used all 50 actions."]
        │   Input still enabled for questions.
        │   GitHub-triggered plans are blocked.
        │
        └── OK
              │
              ▼
        [Bot typing indicator appears]
        "[Bot Name] is thinking..."
              │
              ▼
        Bot response streams via SSE
              │
              ├── Plain text response (no GitHub action)
              │         │
              │         ▼
              │   [Message appears in thread]
              │   Flow ends.
              │
              └── GitHub action required
                        │
                        ▼
                  [Plan card appears inline]
                  ╔══════════════════╗
                  ║  📋 [Bot]'s plan ║
                  ║  [summary]       ║
                  ║  [Approve] [Not] ║
                  ╚══════════════════╝
                        │
                        ├── "Not now" → plan card collapses
                        │   Bot says: "No problem. Let me know when you're ready."
                        │
                        └── "Approve" → [Plan Approval Modal opens]
                                  │
                                  ▼
                          [See Flow 3: Plan Approval]
```

## Flow 3: Plan Approval (The Trust Moment)

**Entry**: Founder clicks "Approve" on a plan card.  
**Exit**: Plan executed; bot posts result. Or: plan rejected.

```
[Founder clicks "Approve" on plan card]
        │
        ▼
[Plan Approval Modal opens]
  - Bot name + "is about to do something"
  - Numbered plain-English action list
  - Action cost shown ("This will use 1 action")
  - Current usage shown ("12 used of 50 this month")
        │
        ├── "Not now" (secondary)
        │         │
        │         ▼
        │   Modal closes. Plan card shows "Not now" state.
        │   Bot posts: "No problem — just say the word when you're ready."
        │   Flow ends.
        │
        └── "Yes, go ahead" (primary)
                  │
                  ▼
            Modal closes.
            Plan card shows loading state: "Sam is working on it..."
                  │
                  ▼
            [Background: POST /api/plans/[id]/approve]
            [GitHub actions execute]
                  │
                  ├── Success
                  │         │
                  │         ▼
                  │   Plan card updates:
                  │   "✓ Sam reviewed PR #42 · comment posted · just now"
                  │   Bot posts follow-up message with findings.
                  │
                  └── Failure
                            │
                            ▼
                      Plan card updates:
                      "Something went wrong — Sam couldn't complete this."
                      Bot posts: "I hit a snag. [plain-English reason]. 
                                  Want me to try a different approach?"
```

## Flow 4: GitHub Trigger (Automatic Bot Response)

**Entry**: GitHub event fires (e.g. PR opened, issue labeled "security").  
**Exit**: Relevant bot has posted a message in the relevant channel.

```
[GitHub event: pull_request.opened]
        │
        ▼
[POST /api/github/webhook]
  HMAC validated
        │
        ▼
[getTriggerRules → match event to channel + bot]
  Example: PR opened → #engineering → Sam (Engineering)
        │
        ▼
[System message posted to #engineering]
  "🔀 New PR: 'Add user auth' by @founder"
        │
        ▼
[Bot auto-responds]
  "I'll take a look at this. Give me a moment."
  [Bot reads PR via GitHub API]
        │
        ▼
  Bot proposes plan if action required (see Flow 3)
  OR posts plain review comment (no plan needed)
        │
        ▼
[Unread badge appears on #engineering in sidebar]
[Push notification sent if founder is away]
```

## Flow 5: Hire a New Teammate

**Entry**: Founder clicks "+ Hire teammate" in sidebar.  
**Exit**: New channel created, new bot posts intro message.

```
[Founder clicks "+ Hire teammate"]
        │
        ▼
[Hire Teammate Modal]
  Select role → pick from available (already-hired greyed out)
        │
        ▼
  Name input (pre-filled suggestion)
        │
        ▼
  "Add [Name] to team" CTA
        │
        ▼
[Modal closes]
[New channel #[role] appears in sidebar]
[Channel is auto-selected]
[New bot posts intro / empty state message]
```

## Flow 6: Standup Room (Scheduled)

**Entry**: Scheduled time (e.g. 9am daily).  
**Exit**: Each bot has posted their standup update.

```
[Vercel Cron fires at 9am]
        │
        ▼
[POST /api/bots/standup trigger]
        │
        ▼
[Each bot posts to #standup]
  "[Bot Name] — [Role]
   Yesterday: [what was done]
   Today: [what's planned]
   Blockers: [any blockers or 'None']"
        │
        ▼
[Unread badge on #standup]
[Founder can reply to any bot in standup thread]
```

## Edge Cases

| Scenario | Handling |
|---|---|
| Founder sends message while bot is still typing | Queued; bot finishes current response first; input disabled during bot turn |
| Founder closes modal mid-approval | Plan stays `pending`; card re-shows Approve/Not now on reopen |
| GitHub not connected; bot proposes GitHub plan | Plan card shows: "Connect GitHub first to let Sam do this" with Connect button |
| Multiple channels with unread messages | All show unread dots; founder decides where to go |
| Bot fails to respond (timeout) | "Sam got stuck. Try again?" with retry button |
| Two bots needed (e.g. security + engineering) | First bot responds; if it needs to loop in another, it says "I'll ask Jordan to weigh in" and posts in Jordan's channel |
