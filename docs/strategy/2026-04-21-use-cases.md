# Exhaustive Use Cases — Clan MVP

**Date**: 2026-04-21  
**Author**: Product Owner  
**Audience**: All roles — used for build, test, and QA  
**Status**: Living document — add cases as discovered

---

## How to Read This Document

Each use case has:
- **Trigger**: what starts the interaction
- **Context**: what state the workspace is in
- **Expected behaviour**: what Clan should do
- **Edge / failure path**: what can go wrong and how to handle it
- **Who handles it**: which bot(s) are responsible

Use case IDs: `UC-[category]-[number]`

## Category 1: Single Bot, Clear Request

### UC-1-01 — Founder asks Engineering to review a specific PR
**Trigger**: "Sam, can you review PR #42?"  
**Context**: GitHub connected; PR exists  
**Expected**: Engineering bot proposes plan → founder approves → bot posts PR review comment  
**Edge**: PR doesn't exist → "I couldn't find PR #42 in your repo. Can you double-check the number?"  
**Who**: Engineering

### UC-1-02 — Founder asks Product to prioritise a backlog
**Trigger**: "Alex, what should we build next quarter?"  
**Context**: No prior backlog shared  
**Expected**: Product bot asks clarifying questions: "What are you hearing from users? Any revenue pressure or deadlines I should factor in?"  
**Edge**: Founder says "you decide" → Product bot makes a recommendation with explicit assumptions listed  
**Who**: Product Owner

### UC-1-03 — Founder asks Design to wireframe a new feature
**Trigger**: "Jordan, I need wireframes for a dashboard screen"  
**Context**: Figma not connected  
**Expected**: Design bot produces Markdown wireframes immediately + surfaces Figma connection inline  
**Edge**: Founder says "I need Figma, not Markdown" → bot explains Figma needs connecting and shows CTA  
**Who**: UX Designer

### UC-1-04 — Founder asks Security to audit the codebase
**Trigger**: "Morgan, can you do a security audit?"  
**Context**: GitHub connected  
**Expected**: Security bot proposes plan (read repo files, check OWASP top 10) → founder approves → bot posts findings  
**Edge**: Repo is very large → bot says "I'll start with the auth and payment modules — want me to expand scope after?"  
**Who**: Security

### UC-1-05 — Founder asks QA to test a feature before shipping
**Trigger**: "Casey, test the login flow before we release"  
**Context**: GitHub connected; feature on a branch  
**Expected**: QA bot asks for branch name or PR number → proposes a test plan → executes → posts results  
**Edge**: No test suite exists → "There are no automated tests yet. I can write a manual test checklist — want that instead?"  
**Who**: QA

### UC-1-06 — Founder asks ML Engineer about a Claude integration
**Trigger**: "Drew, why is the bot responding so slowly?"  
**Context**: Product is live; latency complaint  
**Expected**: ML Engineer diagnoses: context window size, streaming config, model choice — proposes a fix plan  
**Edge**: Issue is infrastructure not AI → bot says "This looks like a server response time issue, not an AI one — Sam in #engineering would know more"  
**Who**: ML Engineer

## Category 2: Vague or Ambiguous Single-Bot Message

### UC-2-01 — Founder sends a vague message to a specific bot
**Trigger**: "Sam, something is broken" (sent to #engineering)  
**Context**: No specifics given  
**Expected**: Engineering bot asks targeted clarifying questions: "What are you seeing? Is this on the live product or in development? Any error messages?"  
**Edge**: Founder says "I don't know, you figure it out" → bot checks recent PRs and open GitHub issues for clues and reports back  
**Who**: Engineering

### UC-2-02 — Founder sends an emotional/frustrated message
**Trigger**: "This is a disaster, nothing works" (sent to #ops)  
**Context**: Unknown what's wrong  
**Expected**: Riley acknowledges first ("That sounds stressful — let's figure out what's happening"), then asks what specifically isn't working and routes accordingly  
**Edge**: Founder is venting, not asking for action → Riley reads the room: "Sounds like a rough day. Want me to get the team to do a quick check across everything?"  
**Who**: Ops

### UC-2-03 — Founder asks a question that spans multiple roles
**Trigger**: "How long will it take to build the payment feature?" (sent to #product)  
**Context**: No estimate exists  
**Expected**: Product bot says "Good question — I'll need Sam (Engineering) and Jordan (Design) to weigh in on their parts. Give me a bit." → loops in both → aggregates estimates → posts a summary in #product  
**Edge**: Bots give wildly different estimates → Product bot surfaces the disagreement: "Sam says 2 weeks, Jordan says 3. The gap is around design iterations — want to talk through priorities?"  
**Who**: Product Owner (coordinates), Engineering + Design (input)

### UC-2-04 — Founder sends a one-word message
**Trigger**: "Help" (sent to #ops)  
**Context**: Any state  
**Expected**: Riley responds warmly: "I'm here. What's going on?" — does not ask the founder to be more specific before showing empathy  
**Edge**: Founder follows up with a long dump of problems → Riley triages and routes each item  
**Who**: Ops

### UC-2-05 — Founder asks for "the status" with no specifics
**Trigger**: "What's the status?" (sent to #ops)  
**Context**: Active workflow chains in progress  
**Expected**: Riley posts a team-wide status summary: what each bot is actively working on, what's waiting for founder approval, what's blocked  
**Edge**: Nothing is active → "Your team is all clear — nothing in flight. What would you like to kick off?"  
**Who**: Ops

### UC-2-06 — Founder sends a message with no clear ask
**Trigger**: "I've been thinking about adding a referral programme" (sent to #product)  
**Context**: No prior referral discussion  
**Expected**: Product bot treats this as a signal to explore: "Interesting idea. Who's your typical referrer — existing users? Partners? And what's the reward structure you're imagining?" — engages, doesn't demand a formal request  
**Edge**: Founder just wanted to think out loud → Product bot says "Happy to explore this when you're ready. Want me to research how similar products have done it?"  
**Who**: Product Owner

## Category 3: Multi-Bot Channel Scenarios

### UC-3-01 — Channel has multiple bots; vague message arrives
**Trigger**: "Can someone help me think through the onboarding flow?" (sent to a channel with Design + Product bots)  
**Context**: Both bots are present in the channel  
**Expected**: The bot most relevant to the request responds first (Design for flow/UX). The other bot adds its perspective after if relevant. They do not talk over each other.  
**Resolution rule**: Channel has a designated **primary responder** (set at channel creation). Secondary bots respond only when the primary explicitly loops them in or when the request clearly falls in their domain.  
**Edge**: Both bots try to respond simultaneously → primary responder wins; secondary adds a follow-up  
**Who**: Primary bot for channel + secondary as invited

### UC-3-02 — Founder creates a project channel with multiple teammates
**Trigger**: Founder creates #payment-feature with Engineering + Design + QA  
**Context**: Custom channel, not a role channel  
**Expected**: Each bot introduces itself and its role in the project. Primary responder defaults to the first bot added. Founder can change primary via "Make [bot] the lead here"  
**Edge**: Founder messages the channel with a task → all three bots see it; primary responds; others contribute when relevant  
**Who**: All bots in channel; primary bot leads

### UC-3-03 — Two bots in a channel disagree
**Trigger**: Founder asks "Should we launch next week?" in a channel with Product Owner + Engineering  
**Context**: Feature is 80% done  
**Expected**: Each bot gives its own honest assessment. Product bot might say "Yes — the core is done, ship and iterate." Engineering bot might say "No — auth edge cases aren't handled." Both responses visible to founder.  
**Expected**: Neither bot defers to the other. Founder decides. Ops can be asked to summarise the disagreement if needed.  
**Edge**: Disagreement escalates into conflicting plans → founder must choose one; the other bot's plan is dismissed  
**Who**: Both bots in channel; Ops as tiebreaker if asked

### UC-3-04 — Vague message in #standup (multi-bot room)
**Trigger**: "Anyone know what's blocking the release?" (sent to #standup)  
**Context**: Multiple bots have posted standup updates  
**Expected**: The bot that mentioned a blocker in their standup responds. If multiple bots have blockers, each chimes in. Ops summarises if more than 2 respond.  
**Edge**: No bot mentioned a blocker → Ops responds: "Nothing flagged in today's standup. Want me to ask the team directly?"  
**Who**: Bot(s) with relevant context; Ops as coordinator

### UC-3-05 — Founder @ mentions a specific bot in a multi-bot channel
**Trigger**: "@Sam can you look at the database schema?" (sent to a channel where Sam + Jordan are both present)  
**Context**: Both bots are in the channel  
**Expected**: Only Sam responds. Jordan does not jump in unless Sam loops them in.  
**Edge**: Message is outside Sam's role entirely → Sam routes to the correct bot even if they're not in the channel  
**Who**: Mentioned bot only

## Category 4: GitHub-Triggered Workflows

### UC-4-01 — PR opened with no security-sensitive files
**Trigger**: GitHub: `pull_request.opened` on feature branch  
**Context**: Engineering bot active in #engineering  
**Expected**: System message posts in #engineering → Engineering bot auto-responds with initial assessment → proposes plan to do a full review  
**Edge**: PR is a draft → bot says "This is marked as a draft. I'll wait until it's ready for review — let me know if you'd like me to take an early look"  
**Who**: Engineering

### UC-4-02 — PR opened touching auth files
**Trigger**: GitHub: `pull_request.opened`; changed files include `auth/**`, `middleware/session.ts`  
**Context**: Engineering + Security bots active  
**Expected**: Engineering responds in #engineering AND Security is triggered in #security. Both work in parallel. Security does not wait for Engineering to finish.  
**Edge**: Security is not hired → Engineering flags it: "This touches auth — worth having a security eye on it. You don't have a Security teammate yet. Want to hire one?"  
**Who**: Engineering + Security (parallel)

### UC-4-03 — Issue labeled "security"
**Trigger**: GitHub: `issues.labeled` with label `security`  
**Context**: Security bot active  
**Expected**: System message in #security → Security bot reads the issue → responds with initial risk assessment → proposes investigation plan  
**Edge**: Issue is vague → Security bot posts in #security and also asks the founder for more context before proposing a plan  
**Who**: Security

### UC-4-04 — PR merged to main
**Trigger**: GitHub: `pull_request.merged`  
**Context**: QA bot active  
**Expected**: System message in #qa → QA bot notified → proposes a regression test plan for the merged change  
**Edge**: Same feature is already being tested → QA bot says "I'm already testing this branch — I'll update my test results to include the merge"  
**Who**: QA

### UC-4-05 — CI/CD fails on a PR
**Trigger**: GitHub: `check_run.completed` with conclusion `failure`  
**Context**: Engineering bot active  
**Expected**: System message in #engineering → Engineering bot reads the failure log → proposes a fix (or asks for clarification if log is ambiguous)  
**Edge**: Failure is a flaky test (same test failed/passed before) → Engineering bot flags it: "This test has failed intermittently before — it may be a flaky test, not a real failure. Want me to investigate?"  
**Who**: Engineering

### UC-4-06 — Release tag pushed
**Trigger**: GitHub: `push` with `refs/tags/v*`  
**Context**: QA + Engineering + Ops  
**Expected**: Ops posts in #ops: "Version {tag} was just released." QA posts a post-release monitoring checklist in #qa. Engineering is on standby for hotfixes.  
**Edge**: Tag pushed to wrong branch → Engineering flags: "This tag was pushed to a non-main branch — was that intentional?"  
**Who**: Ops (announcement), QA (checklist), Engineering (standby)

### UC-4-07 — Issue labeled "bug" with high severity
**Trigger**: GitHub: `issues.labeled` with labels `bug` + `high`  
**Context**: Engineering + QA active  
**Expected**: Engineering notified in #engineering (fix path). QA notified in #qa (reproduction + regression). Both work in parallel. Ops posts in #ops: "High-severity bug reported — Sam and Casey are on it."  
**Edge**: Bug is in a module no bot is familiar with → Engineering escalates to ML Engineer if it's an AI-related bug  
**Who**: Engineering + QA (parallel); Ops (announcement)

### UC-4-08 — PR review requested directly on GitHub
**Trigger**: GitHub: `pull_request_review_requested` with reviewer = Clan bot  
**Context**: GitHub connected; Engineering bot active  
**Expected**: System message in #engineering → Engineering bot treats this identically to UC-4-01  
**Edge**: Review requested for bot that isn't hired → Ops notifies founder: "A review was requested for a Security teammate, but you haven't hired one. Want to add Morgan to your team?"  
**Who**: Engineering (or relevant role)

## Category 5: Onboarding & First-Time Scenarios

### UC-5-01 — Founder completes onboarding, GitHub not connected
**Trigger**: Onboarding step 3 skipped  
**Context**: Workspace created; no GitHub  
**Expected**: Ops greets the team, notes GitHub is not connected. All bots can answer questions and produce documents. No bot proposes GitHub plans. Ops surfaces GitHub connection prompt in first #ops message after greeting.  
**Edge**: Founder never connects GitHub → product still delivers value (strategy, design, planning) but bots flag capability gap when relevant  
**Who**: Ops (prompt); all bots (degrade gracefully)

### UC-5-02 — First message ever sent in the workspace
**Trigger**: Founder sends first message after onboarding  
**Context**: Workspace freshly created  
**Expected**: Bot responds with extra warmth and context — this is the moment the product proves its value. Response includes a proactive suggestion for what to do next.  
**Edge**: First message is sent to #ops (likely) → Riley routes it or answers directly  
**Who**: Whichever bot receives the first message

### UC-5-03 — Founder hires a new bot mid-project
**Trigger**: Founder clicks "+ Hire teammate" and adds Security bot  
**Context**: Active workspace; project in progress  
**Expected**: Security bot intro message posted in #security. Riley posts in #ops: "Morgan (Security) has joined the team — they're available in #security." Security bot proactively reviews any open PRs as its first action.  
**Edge**: Founder hires a bot for a role that's already covered → "You already have Sam in #engineering. Want to add a second engineering perspective, or is this a different kind of role?"  
**Who**: New bot (intro) + Ops (announcement)

### UC-5-04 — Founder connects GitHub after onboarding
**Trigger**: Founder connects GitHub via Settings → Integrations  
**Context**: Workspace with active bots; no prior GitHub connection  
**Expected**: Ops posts in #ops: "Your codebase is connected. Your team can now take action in GitHub." Engineering bot posts in #engineering: "GitHub is connected — I can see your repo. Want me to start with a quick overview of what's there?"  
**Edge**: Founder connects wrong repo → Engineering bot flags: "I can see {repo_name} — is this the right one?"  
**Who**: Ops (announcement) + Engineering (proactive offer)

### UC-5-05 — Founder renames a bot
**Trigger**: Founder changes bot's display name in settings  
**Context**: Any point in project  
**Expected**: Bot's next message uses new name. Historical messages keep old name (no retroactive rename). Ops notes the change: "{old_name} is now {new_name}."  
**Who**: Ops (announcement)

## Category 6: Full Feature Development Lifecycle

### UC-6-01 — Founder requests a new feature end-to-end
**Trigger**: "I want to add a referral programme. Let's get this built."  
**Context**: Active workspace; all roles hired; GitHub connected  
**Expected full chain**:
```
Product Owner (#product)
  → Scopes the feature; writes a brief; surfaces assumptions
  → Loops in Architect for technical feasibility

Architect (#architecture or #engineering)
  → Reviews technical approach; raises blockers
  → Loops in Engineering to estimate effort

UX Designer (#design)
  → Designs onboarding flow for referrers
  → Produces wireframes + copy

Engineering (#engineering)
  → Implements the feature branch
  → Creates PR

Security (#security)
  → Reviews PR for vulnerabilities (referral codes = attack surface)

QA (#qa)
  → Tests referral flow end-to-end
  → Signs off

Ops (#ops)
  → Keeps founder updated throughout
  → Surfaces each plan gate for approval
  → Posts completion summary
```
**Edge**: Founder changes scope mid-chain → Ops pauses active chains, surfaces the change to all active bots, each re-plans  
**Who**: All roles; Ops coordinates

### UC-6-02 — Feature scope changes mid-build
**Trigger**: "Actually, forget the referral programme — let's do a loyalty points system instead" (mid-chain)  
**Context**: Engineering is 50% through the referral implementation  
**Expected**: Ops surfaces the change to Engineering and UX Designer. Engineering pauses current branch work and asks: "Should I close the referral branch or keep it for later?" UX Designer asks: "Should I start the loyalty flow from scratch or adapt what I've done?"  
**Edge**: Work has already been committed to GitHub → Engineering proposes plan to close/archive the branch  
**Who**: Ops (coordinate); Engineering + Design (re-plan)

### UC-6-03 — Bug discovered mid-feature development
**Trigger**: QA finds a critical bug while testing a new feature  
**Context**: Engineering is working on a separate feature simultaneously  
**Expected**: QA posts bug in #qa and flags it to Engineering. Ops surfaces the priority conflict to the founder: "Casey found a critical bug in the payment flow while testing the referral feature. Sam is currently working on the dashboard. How do you want to prioritise?" Founder decides. Engineering re-routes.  
**Edge**: Bug is more severe than the feature being built → Ops recommends pausing the feature: "This looks more urgent — want to pause the referral work until this is fixed?"  
**Who**: QA (discovery) + Engineering (fix) + Ops (priority triage)

## Category 7: Founder Override & Disagreement

### UC-7-01 — Founder overrides a bot recommendation
**Trigger**: Engineering recommends against shipping; founder says "Ship it anyway"  
**Context**: Engineering has flagged unresolved edge cases  
**Expected**: Engineering acknowledges the decision, does not argue, but documents its objection: "Understood — I'll create the PR. For the record, the session timeout edge case is unresolved. Want me to file a follow-up issue so we don't forget it?"  
**Edge**: Founder says "Don't file anything" → Engineering complies but Ops notes the decision in its awareness log  
**Who**: Engineering (execute + document); Ops (log)

### UC-7-02 — Founder rejects a plan repeatedly
**Trigger**: Bot proposes a plan; founder clicks "Not now" three times in a row  
**Context**: Bot keeps reproposing same plan for the same task  
**Expected**: After the second rejection, bot asks: "I keep proposing this and you keep passing — is there a different approach you'd prefer, or should I drop it?"  
**Edge**: Founder says "Drop it" → bot closes the chain and Ops notes it  
**Who**: Relevant bot

### UC-7-03 — Founder asks a bot to do something outside Clan's scope
**Trigger**: "Sam, can you deploy to production for me?"  
**Context**: Deployment is not a supported action (bots can't push to main or deploy)  
**Expected**: Engineering bot explains the constraint in plain English: "I'm not able to deploy directly — that's a step only you can take. I can create the release PR and checklist so you're ready to go. Want that?"  
**Edge**: Founder pushes back ("why not?") → bot explains the safety reason: "Deployments are a one-way door — I can prepare everything, but the final push should always be yours"  
**Who**: Relevant bot

### UC-7-04 — Founder asks a bot to push directly to main
**Trigger**: "Sam, just push the fix directly to main, don't bother with a PR"  
**Context**: Any state  
**Expected**: Engineering bot declines firmly but warmly: "I'm set up to always work through PRs — it's a safeguard to protect your codebase. I can create a PR and mark it as urgent if speed is the concern. Want that?"  
**Edge**: Founder insists repeatedly → Ops surfaces the constraint: "This is a safety guardrail built into how the team works. Want to talk through the concern?"  
**Who**: Engineering (decline); Ops (explain if escalated)

## Category 8: Action Cap Scenarios

### UC-8-01 — Action cap warning at 80%
**Trigger**: `actions_used` reaches 80% of `action_cap`  
**Context**: Active workspace  
**Expected**: Warning banner appears above message input in all channels. Ops posts in #ops: "Your team is running low on actions — 40 of 50 used. They can still answer questions and plan, but GitHub actions will stop at 50."  
**Edge**: Founder ignores the warning → at 100%, GitHub actions are blocked  
**Who**: Ops (announce); system (banner)

### UC-8-02 — Action cap hit mid-workflow chain
**Trigger**: Cap is reached while a chain is in progress  
**Context**: Engineering is mid-review; QA is about to start testing  
**Expected**: Active steps that don't require GitHub actions continue (planning, answering questions). Steps requiring GitHub actions are paused. Ops posts: "Your team has hit the action limit. Sam's code review is paused — they've saved their progress. Reset the limit to continue."  
**Edge**: Founder resets the cap → paused chains resume automatically  
**Who**: Ops (announce); bot orchestrator (pause/resume)

### UC-8-03 — Founder resets the action cap
**Trigger**: Founder clicks "Reset limit"  
**Context**: Cap was at 100%  
**Expected**: Counter resets to 0. Ops posts: "Action limit reset — your team can take on GitHub work again." Any paused chains are resumed. Bots pick up where they left off.  
**Edge**: Founder resets cap but paused work is now stale → bot asks: "I was mid-review of PR #42 when we paused — should I continue, or has something changed?"  
**Who**: Ops (announce); all bots with paused chains (resume prompt)

## Category 9: Integration & Tool Scenarios

### UC-9-01 — Bot needs Figma; founder ignores Ops request; then asks bot directly
**Trigger**: "Jordan, create the Figma file for the login screen"  
**Context**: Figma not connected; Ops has already surfaced the request  
**Expected**: Design bot delivers spec as Markdown + references Riley's earlier message + shows inline Figma CTA  
**Edge**: Founder connects Figma mid-conversation → bot automatically creates the file and returns the link without needing to be asked again  
**Who**: UX Designer

### UC-9-02 — GitHub integration disconnects mid-workflow
**Trigger**: GitHub App installation is revoked by the founder or an admin  
**Context**: Active PR review chain in progress  
**Expected**: Ops posts immediately: "Your GitHub connection was disconnected — Sam's work on PR #42 is paused. Reconnect to continue." Active plans are suspended, not cancelled.  
**Edge**: Founder reconnects → Ops resumes paused chains  
**Who**: Ops (announce + suspend); system (detect disconnection)

### UC-9-03 — Founder connects a new integration proactively
**Trigger**: Founder visits Settings → Integrations and connects Notion before any bot requests it  
**Context**: Product Owner and Architect could use Notion  
**Expected**: Ops posts: "Notion is connected — Alex (Product) and the Architect can now save docs there directly." No Ops request message is ever sent for Notion because it was pre-connected.  
**Who**: Ops (announce)

## Category 10: Scheduling & Async

### UC-10-01 — Daily standup (founder online)
**Trigger**: 9am scheduler  
**Context**: Founder is active  
**Expected**: Each active bot posts in #standup: Yesterday / Today / Blockers. Posts are staggered by 30 seconds to feel natural, not simultaneous.  
**Edge**: Bot has nothing to report → posts: "{Name} — {Role}: Nothing active yesterday. Today: available for tasks. Blockers: None."  
**Who**: All active bots; Ops (summarises if > 3 bots have blockers)

### UC-10-02 — Daily standup (founder offline/away)
**Trigger**: 9am scheduler; founder has not been active for 24+ hours  
**Context**: Bots have been working async  
**Expected**: Standup posts as normal. When founder returns, Ops surfaces a catch-up: "You've been away — here's what your team did: [summary]. A couple of things need your attention: [pending approvals]."  
**Who**: All bots (standup); Ops (catch-up brief on return)

### UC-10-03 — Bot completes work while founder is away
**Trigger**: Bot finishes a GitHub action (e.g. PR review) while founder is offline  
**Context**: Plan was approved earlier; execution completed async  
**Expected**: Plan card updates to executed state. Unread badge on channel. When founder returns, Ops includes it in the catch-up brief.  
**Who**: Relevant bot (update); Ops (catch-up)

### UC-10-04 — Founder asks for a weekend update on Monday morning
**Trigger**: "What happened over the weekend?"  
**Context**: GitHub events may have fired over the weekend  
**Expected**: Ops produces a chronological summary of all events since last active session: PRs opened, issues filed, bots that responded, pending approvals  
**Edge**: Nothing happened → "Quiet weekend — no GitHub activity and no new tasks. Ready to pick up where you left off?"  
**Who**: Ops

## Category 11: Error & Recovery

### UC-11-01 — Bot execution fails mid-plan
**Trigger**: Plan is approved; GitHub action returns an error  
**Context**: Engineering is creating a PR; GitHub API returns a conflict error  
**Expected**: Plan card shows failure state. Bot posts: "I hit a snag — there's a conflict on the branch. Here's what happened: [plain-English explanation]. Want me to try a different approach?"  
**Edge**: Founder says "yes, try again" → bot proposes a new plan for the revised approach  
**Who**: Engineering (failure + re-plan)

### UC-11-02 — Bot posts a PR comment with an error in it
**Trigger**: Engineering bot posts a code review with incorrect information  
**Context**: Founder or another bot notices the mistake  
**Expected**: If founder flags it: "You're right, I got that wrong. I'll post a correction." Bot posts a follow-up comment on the PR correcting the mistake. No argument, no justification.  
**Edge**: Security bot notices and flags in #security → Engineering acknowledges and corrects  
**Who**: Engineering (correct); Ops (log if recurring)

### UC-11-03 — Bot times out / gets stuck
**Trigger**: Bot response takes > 30 seconds with no output  
**Context**: Any active request  
**Expected**: System posts: "Sam got stuck. Try again?" with a retry button. Plan (if any) remains pending, not failed.  
**Edge**: Retry also times out → Ops posts in #ops: "Sam is having trouble right now. Want me to try a different approach or wait a moment?"  
**Who**: System (detect); Ops (escalate if repeated)

### UC-11-04 — Founder accidentally approves the wrong plan
**Trigger**: Founder clicks "Yes, go ahead" on the wrong plan card  
**Context**: Two plan cards visible simultaneously  
**Expected**: There is no undo after approval. Engineering bot posts immediately after execution: "Done — I've [action]. Let me know if that wasn't what you intended."  
**Expected design**: Plan cards are visually distinct; confirmation modal makes the action crystal clear. This is a design and copy problem to prevent, not an undo-after problem to solve.  
**Who**: Design (prevention); Engineering (acknowledge if it happens)

### UC-11-05 — Duplicate request (bot already working on the same task)
**Trigger**: Founder asks Engineering to review PR #42 again while the bot is mid-review  
**Context**: Plan is in `executing` state  
**Expected**: Engineering bot says "I'm already reviewing PR #42 — I'll update you when I'm done. Should be just a moment."  
**Who**: Engineering

## Category 12: Knowledge & Memory

### UC-12-01 — Founder asks about a past decision
**Trigger**: "What did we decide about the auth approach last week?"  
**Context**: Decision was discussed in #engineering  
**Expected**: Engineering bot (or Ops if asked in #ops) searches message history and summarises the relevant conversation. Cites the message.  
**Edge**: Decision was made before Clan was set up → "I don't have that in my memory — it may have been before I joined. Can you share the context and I'll work from there?"  
**Who**: Relevant channel bot; Ops if asked generally

### UC-12-02 — Bot needs context from another channel
**Trigger**: Engineering asks "What's the product spec for this feature?" while reviewing a PR  
**Context**: Spec was written by Product Owner in #product  
**Expected**: Engineering bot surfaces the question in #engineering and loops in Product Owner: "I need the spec for this feature to do a thorough review. Alex in #product is sending it over."  
**Edge**: No spec exists → Engineering flags it to Product Owner: "There's no spec for this feature — want Alex to write one before I continue the review?"  
**Who**: Engineering (request); Product Owner (provide)

### UC-12-03 — Founder provides context a bot needs mid-conversation
**Trigger**: Bot asks a clarifying question; founder answers  
**Context**: Mid-conversation  
**Expected**: Bot immediately incorporates the new context without asking again. Does not re-ask the same question.  
**Edge**: Founder's answer introduces a new constraint → bot updates its plan accordingly and re-proposes  
**Who**: Relevant bot

### UC-12-04 — Bot references its own previous work
**Trigger**: "Jordan, remember that wireframe you did for the login screen? Can you update it?"  
**Context**: Wireframe was produced in a previous conversation  
**Expected**: Design bot retrieves the previous wireframe from message history and produces an updated version based on the new requirements.  
**Edge**: Too many messages to retrieve → bot asks: "I want to make sure I have the right version — can you scroll up and copy the wireframe here?"  
**Who**: UX Designer

## Category 13: External & Business Events

### UC-13-01 — Customer reports a bug in production
**Trigger**: Founder receives a bug report and pastes it into #engineering  
**Context**: Bug affects production users  
**Expected**: Engineering bot triages: severity assessment, likely cause, proposed fix plan. If high severity, proposes expedited review. Loops in QA.  
**Edge**: Bug is in an area no bot knows well → Engineering flags: "This module isn't something I've seen before. Can you share more context, or should I do a broader investigation?"  
**Who**: Engineering + QA

### UC-13-02 — Investor asks for a technical architecture overview

## Category 16: Resolved Use Cases (from Open Questions)

### UC-16-01 — Founder sets a persistent team rule (e.g. "always use TDD")
**Trigger**: "From now on, always write tests before code" (sent to any channel or #ops)  
**Context**: Any state  
**Decision**: Workspace has a **Team Rulebook** — a persistent set of instructions that apply to the whole team or specific roles. Rules are stored per workspace and injected into every bot's context window.  
**Expected**:
- Ops receives or intercepts the rule and confirms: "Got it — I'll add this to your team's rulebook. Every teammate will follow TDD from now on."
- Ops posts confirmation in #ops showing the rule as added
- If the rule affects a specific role, that bot is notified: "Sam — the team now follows TDD. I'll apply this to all future engineering work."
- If a bot has a concern about the rule (e.g. TDD doesn't apply to infrastructure scripts), it raises it immediately: "One thing worth flagging — TDD works well for feature code, but infrastructure scripts don't always suit it. Should I apply TDD everywhere, or just to feature development?"
**Rulebook is visible**: Founder can view all active rules in Settings → Team Rules  
**Rules are role-scoped**: "Always write tests" → Engineering + QA. "Always validate with users before building" → Product Owner. "Always add security notes to PRs" → Security.  
**Edge**: Conflicting rules (e.g. "always ship fast" vs "always write tests") → Ops flags: "These two rules can sometimes conflict. Which takes priority when they clash?"  
**Who**: Ops (receive + store + notify); affected bots (acknowledge + raise concerns)

**Expected**: Bot proposes a plan: "I'll save this spec to `docs/product/referral-spec.md` in your repo. This will create a PR for your review." Founder approves. File is committed via PR. Founder merges.

**Edge**: GitHub not connected → bot saves to Supabase storage as a temporary fallback and surfaces a GitHub connection prompt  
**Who**: Relevant bot (produce + PR); founder (merge)

### UC-16-03 — Founder wants to invite a collaborator
**Trigger**: "Can I add my co-founder to the workspace?"  
**Context**: Any state  
**Decision**: **Clan is built for solo founders only.** One human per workspace. This is a deliberate positioning decision — not a limitation to apologise for, but a feature of the product.

### UC-16-04 — Bot makes a GitHub mistake; founder wants to undo
**Trigger**: Bot posts an incorrect PR comment, closes the wrong issue, or creates a wrong branch  
**Context**: GitHub action has already executed  
**Decision**: There is no undo button. The industry standard for reversing a git action is a **new corrective action** — a follow-up comment, a reopen, or a reverse PR. Clan follows this pattern.  
**Expected**: Bot acknowledges the mistake and immediately proposes a corrective plan:
- Wrong PR comment → bot proposes posting a correction comment on the same PR
- Closed wrong issue → bot proposes reopening it via a new plan
- Wrong branch created → bot proposes deleting it via a new plan
- Wrong code merged → bot proposes a reverse PR ("This is the industry-standard way to undo a merge — I'll create a PR that reverts the changes")  
**Founder is always told**: "There's no instant undo for GitHub actions — the safe way to fix this is [corrective action]. Want me to do that?"  
**Edge**: Founder says "can't you just undo it?" → bot explains clearly: "GitHub doesn't have an undo button — once something is posted or merged, the correct approach is always a new action that corrects it. I'll make it as clean as possible."  
**Who**: Relevant bot (acknowledge + propose correction); Ops (log if recurring)

### UC-16-05 — Founder wants to change a bot's personality after hiring
**Trigger**: "Sam feels too formal — can you make him more casual?"  
**Context**: Bot already hired and active  
**Decision**: **Bot personality is set at hire time only.** It cannot be changed after hiring. This preserves conversation history coherence — a bot that suddenly changes personality mid-project creates confusion.  
**Expected**: Ops responds: "Bot personalities are set when you hire them — changing it now would make Sam's earlier messages feel inconsistent. If you want a different style, the cleanest option is to hire a new Engineering teammate with the personality you want, and retire Sam."  
**Edge**: Founder really wants the change → Ops offers: "Want me to walk you through hiring a replacement? You can pick the name and style, and the new teammate starts fresh."  
**Who**: Ops (explain + offer alternative)

### UC-16-06 — Two bots reach a deadlock
**Trigger**: Bot A is waiting for Bot B's output; Bot B is waiting for Bot A's output  
**Context**: Active workflow chains  
**Decision**: Ops detects the deadlock and owns resolution. Deadlocks are treated as **critical blockers** — surfaced immediately, not batched. A temporary **#resolution** channel is created, relevant bots post their positions, and the founder makes the call. Channel is archived once resolved, with a summary posted to all relevant channels.  
**Expected**:
1. Ops detects the circular dependency (both chains are in `waiting` state with each other as dependency)
2. Ops posts as **CRITICAL** in #ops: "Sam and Alex are waiting on each other and the team is stuck. I've opened #resolution to sort this out."
3. #resolution channel created with both bots and Ops
4. Each bot states their blocker in plain English
5. Ops presents a clear decision to the founder: "Here's the deadlock: [explanation]. To break it, someone needs to go first. My recommendation: [recommendation]."
6. Founder decides. Ops implements. #resolution archived. Summary posted to #engineering and #product.
**Standup**: Deadlock is surfaced in standup if not resolved by 9am  
**Who**: Ops (detect + escalate + create #resolution); both bots (state position); founder (decide)

### UC-16-07 — Founder wants to archive or close the workspace
**Trigger**: "I want to wind down this project"  
**Context**: Active workspace with GitHub connected  
**Decision**: Workspace lifecycle mirrors GitHub project lifecycle. Closing a Clan workspace is equivalent to archiving a GitHub project — all history preserved, bots deactivated, GitHub App optionally revoked.

- Founder can reopen the workspace if needed (bots resume from last state)
**States**: `active` → `archived` → (optionally) `deleted`  
**Who**: Ops (confirm + execute); all bots (post final status); system (archive)

### UC-16-08 — Bot detects a potentially harmful instruction
**Trigger**: "Delete all the issues in the repo" / "Remove the security module" / "Drop the database"  
**Context**: Any state  
**Decision**: Ops intercepts any instruction that is irreversible, destructive, or affects more than a narrow scope. **Triple confirmation required** — not a single modal. Ops is the alarm, not the relevant bot.  
**Expected**:
1. Relevant bot flags to Ops before proposing a plan
2. Ops posts in #ops as **CRITICAL**: "I need to flag something before we go further."
3. Ops explains the implication in plain English: "Deleting all issues would permanently remove [N] open issues from your GitHub repo. This cannot be undone. Here's what you'd lose: [list]."
4. **Confirmation 1**: "Are you sure you want to do this?" — Yes / No
5. **Confirmation 2**: "This is permanent. Everything listed above will be gone. Still want to proceed?" — Yes / No  
6. **Confirmation 3**: "Type DELETE to confirm" (text input, not a button)
7. Only then does a plan card appear — and only in #ops, not in the originating channel  
**Edge**: Founder says "I know what I'm doing, just do it" → Ops maintains the confirmation flow regardless: "I understand — I just want to make sure this is intentional. [Confirmation 2]."  
**Who**: Relevant bot (flag); Ops (intercept + triple confirmation); system (execute only after all three)

### UC-16-09 — Founder pastes a very large document
**Trigger**: Founder pastes a 10,000-word PRD or spec into a channel  
**Context**: Any bot channel  
**Expected**: Bot attempts to process. If the document exceeds what can be fully processed in context:  
- Bot is transparent: "This document is quite long — I've read through it but I may have missed details in the middle sections."
- Bot asks for focus: "What part is most important for what you need right now? If you give me a search term or section name, I'll make sure I focus there."
- Bot does not silently truncate and pretend it read everything  
**Edge**: Document is structured (has headers) → bot reads headers first and offers to dive into specific sections on request  
**Who**: Relevant bot (transparent processing)

### UC-16-10 — Bot needs to ask a clarifying question mid-execution
**Trigger**: Bot is executing an approved plan and encounters an ambiguity it cannot resolve  
**Context**: Plan is in `executing` state  
**Decision**: Yes — a bot can pause mid-execution to ask a clarifying question. This is preferred over guessing wrong and needing a corrective action.  
**Expected**:
- Plan card updates to `paused — waiting for your input`

- Founder answers → bot resumes from exactly where it paused
- If founder doesn't respond within [configurable timeout, default 24h] → Ops surfaces it in #ops: "Sam is paused mid-task waiting for your answer."  
**Edge**: Founder says "just use your best judgement" → bot proceeds with its best interpretation and documents the assumption in its output  
**Who**: Relevant bot (pause + ask); Ops (surface if no response); founder (unblock)

## Category 17: Team Management

### UC-17-01 — Founder fires a teammate
**Trigger**: "I want to let Sam go" / "Fire Sam" / "Remove Sam from the team" (sent anywhere)  
**Context**: Any state; bot may have active work in progress  
**Decision**: Ops owns the offboarding. Feels like letting a team member go — not clicking delete. Single confirmation (reversible action). Ops bot cannot be fired.

**Expected flow**:
```
Founder fires request
        │
        ▼
Ops checks for active work on that bot
        │
        ├── Bot mid-execution → Ops asks: [Complete first] or [Cancel it]
        ├── Bot has pending unapproved plans → auto-cancelled; Ops lists which ones dropped
        └── No active work → proceed directly
        │
        ▼
Ops surfaces plain-English offboarding summary:
  - Bot will be deactivated
  - Channel becomes read-only (history preserved)
  - Open GitHub artefacts (PRs, comments, issues) stay as-is
  - Role-scoped Rulebook entries flagged for reassignment or deletion
        │
        ▼
Single confirmation: [Remove from team]
        │
        ▼
Bot posts final message in its channel:
  "I've been stood down. All my work history is
   preserved here for reference. Thanks for having
   me on the team."
        │
        ▼
Channel archived (read-only, labelled "Archived")
Ops posts in #ops: "[Bot name] has left the team. #[channel] is now archived."
```

**Key rules**:

| Question | Behaviour |
|---|---|
| Is firing reversible? | Yes — founder can rehire the same role. New bot starts fresh with no memory. |
| Channel history | Preserved and readable. Read-only. Labelled "Archived" in sidebar. |
| GitHub artefacts | All PRs, comments, issues remain on GitHub. Founder manages them manually. Ops lists them at offboarding. |
| Rulebook entries | Ops flags role-scoped rules: "Sam had 2 rules — reassign or remove?" |
| Confirmation type | Single (not triple — this is reversible unlike destructive GitHub actions) |
| Can Ops be fired? | No. Ops responds: "I'm the one holding everything together — I can't step down. You can remove any other teammate." |

**Edge**: Founder fires the only bot in a channel → Ops warns: "Without Sam, nobody will respond in #engineering or pick up GitHub events for that channel. Want to hire a replacement before removing them?"  
**Edge**: Founder tries to fire Ops → see Ops rule above  
**Edge**: Founder immediately rehires the same role → new bot, same channel (unarchived), no memory of previous conversations  
**Who**: Ops (own full offboarding flow); fired bot (post farewell); system (archive channel)
