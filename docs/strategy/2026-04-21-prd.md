## 1. Product Overview

### What is Clan?
A Slack-like workspace where every teammate is an AI bot. Founders hire a self-sustaining AI team, talk to them like Slack colleagues, and bots actually ship work via GitHub triggers. Zero technical setup — pick a team template and go.

### The Name
**Clan** — Hindi for a self-sustaining swarm/horde. Same unexpected energy as "Slack." Signals Indian roots without being Indian-coded globally.

### Primary User (ICP)
Non-technical founder of an early-stage startup (0–5 person team) who:
- Has a GitHub repo but doesn't write code themselves
- Is currently managing contractors or a small dev team via Slack/Notion/email
- Spends more time coordinating work than doing it
- Has no budget or bandwidth for a full-time engineering team

### The First 10 Minutes
Founder clicks a magic-link email → names their company → picks a team template → optionally connects GitHub → meets their team via Ops bot intro → sends their first message. **All within 10 minutes. No technical setup required.**

## 2. Core Product Principles

1. **Feels like hiring, not configuring** — every interaction uses team/people language, never technical language
2. **Plan before action, always** — no bot takes a GitHub action without a founder-approved plan
3. **Trust is built at every interaction** — especially the plan approval modal; clarity over cleverness
4. **Ops is the office manager** — all team admin, integration requests, and blockers route through #ops
5. **Blocked bots degrade gracefully** — partial output + inline unblock CTA, never a hard error

## 3. Non-Negotiable Constraints

| Constraint | Detail |
|---|---|
| No direct pushes to `main` | Every bot code action creates a PR; no bot has write access to `main` |
| Plan-before-action gate | Bot posts plain-English plan; founder approves before any GitHub action executes |
| Action cap enforced | Hard token/action budget per workspace; enforced at middleware layer |
| Action counter always visible | In top bar at all times; never buried in settings |
| No technical language | Banned: webhook, branch, API, token, model, agent, endpoint, SDK. Use: "teammate", role name, plain English |
| Tests required on every PR | No bot can propose a PR without accompanying tests — including Hotfixes. If no test infrastructure exists, setting it up is the first plan before any feature work. |
| CI must pass before merge | Engineering bot cannot propose a merge plan card if CI is failing. Merge gate only appears after CI is green. |
| Coverage cannot decrease | Each PR must maintain or improve existing code coverage. Target: 70%+ overall, 100% on critical paths (auth, payments, data mutations). Coverage drop triggers a gate challenge. |
| CI/CD setup is a prerequisite | Before any feature work begins, a working CI pipeline must exist. Engineering bot checks on first GitHub connection and proposes setup if missing. |
| Use cases required before design | No feature leaves Stage 2 (Requirements) without written use cases covering the happy path, edge cases, and failure paths. Product bot's sign-off on requirements means both are complete. |
| QA tests against use cases | QA bot's Stage 6 sign-off means every in-scope use case has been verified — passed, failed, or explicitly waived with a documented reason. Test plans reference use case IDs, not feature descriptions. |

## 5. User Stories & Acceptance Criteria

### 5.0 Working Style

**US-000**: As a founder, I want to tell Clan how I like to work so the team communicates with me the right way from day one.
- AC: Working style selector appears as a step in onboarding after template selection, before GitHub connect
- AC: Three cards shown: Hands-off, Balanced, Hands-on — each with plain-English description, no jargon
- AC: Balanced is pre-selected as the default; founder can proceed without changing it
- AC: Selection takes one tap; no form fields
- AC: Selected style is displayed in Settings → My Working Style and editable at any time

**US-000b**: As a hands-off founder, I want Ops to handle routine approvals for me so I only see what truly needs my attention.
- AC: In Hands-off mode, Ops auto-approves: PR review comments, filing GitHub issues, standup summaries, saving docs to GitHub
- AC: Ops never auto-approves: PR creation/merge, deletions, destructive actions, actions costing > 3 actions
- AC: Founder receives a daily digest in #ops instead of real-time per-event notifications
- AC: Bot responses use business outcome language ("Feature shipped") not technical detail

**US-000c**: As a hands-on founder, I want to approve every plan before it executes so nothing happens without my knowledge.
- AC: In Hands-on mode, all plans — including informational ones — surface a plan card requiring approval
- AC: Real-time notification for every bot action and completion
- AC: Bots include full technical detail in their responses by default
- AC: No plan is auto-approved regardless of type or cost

**US-000d**: As a founder, I want to change my working style after onboarding without losing any history.
- AC: Settings → My Working Style shows current mode with all three cards
- AC: Changing mode takes effect immediately
- AC: Ops posts in #ops announcing the change to the team: "Your working style has changed to [mode]. The team will now [plain-English description of what changes]."
- AC: No message history is affected by the change

## 10. Project Management Rules (Feature Stage Model)

*Decision 017 — added 2026-04-21*

### 10.1 The Seven Stages

Every feature (or bug fix or task) lives in exactly one stage at any time. Ops tracks the current stage for each active workstream and is the authoritative source of truth.

| Stage | Name | What it means | Gate to leave |
|---|---|---|---|
| 1 | Discovery | Problem validated; exploration underway | Product Owner sign-off |
| 2 | Requirements | Acceptance criteria written and approved | Founder review + PO sign-off |
| 3 | Design | UX flows and component specs approved | UX Designer sign-off (all working styles) |
| 4 | Tech Plan | Architecture and API contracts approved | Architect sign-off |
| 5 | Build | Engineering actively implementing | QA sign-off on completeness |
| 6 | QA | Testing and bug triage complete | QA sign-off (or explicit risk waiver by founder) |
| 7 | Shipped | Merged to main; smoke check passed | Ops marks shipped |

### 10.2 Complexity Tiers

Before entering Stage 1, every workstream is assigned a complexity tier. The tier governs which gates apply and whether they can be skipped.

| Tier | Type | Typical size | Gates that can be skipped |
|---|---|---|---|
| Hotfix | Bug fix; no new surface | Hours | 1, 2, 3, 4 (enters at Stage 5) |
| Small | 1 surface, clear spec | 1–2 days | 1, 4 (must clear 2, 3) |
| Medium | Multi-surface, some unknowns | 3–5 days | None |
| Large | Multi-team, new architecture | 1+ weeks | None (all gates mandatory) |

### 10.3 Non-Negotiable Gates

These gates **can never be waived by any working style or any bot**:

1. **Use cases complete (Stage 2 → 3)**: Product bot cannot sign off on requirements without written use cases covering the full scenario — happy path, edge cases, and failure paths. UX Designer cannot begin design work until use cases exist. No exceptions for any complexity tier.
2. **Feasibility review (Stage 2 → 3, sub-gate)**: Immediately after use cases are signed off, Product bot triggers a 48-hour feasibility review with UX Designer and Architect simultaneously. Both must return Clear or Red Flag (minor/major) before Stage 3 begins. Minor flags are auto-resolved by the team. Major flags are escalated to the founder per working style and block Stage 3 until resolved. Silence after 48h = Clear.
3. **Design sign-off (Stage 3 → 4)**: UX Designer must sign off before any bot writes implementation code. In Hands-off mode, Ops prompts the founder to confirm design approval; it is never auto-approved.
4. **CI/CD setup (before Stage 5)**: A working CI pipeline and test infrastructure must exist before Engineering begins building any feature. If missing, Engineering bot proposes setup as the first plan. No feature work proceeds without this.
5. **Tests on every PR (Stage 5)**: Engineering bot cannot propose a PR plan that does not include tests. A plan with no tests is challenged before it reaches the founder. Applies to all tiers including Hotfix.
6. **CI green before merge (Stage 5 → 6)**: Engineering bot cannot propose a merge plan card if CI is failing. Founder never sees a merge approval for a failing build.
7. **Coverage gate (Stage 5 → 6)**: Coverage must not decrease. If it does, Engineering bot posts a gate challenge. Founder can override with a documented exception — same pattern as other gates.
8. **QA signs off against use cases (Stage 6 → 7)**: QA bot's sign-off means every in-scope use case has been verified — passed, failed, or risk-waived. A feature with unverified use cases cannot advance to Shipped regardless of working style.
9. **Merge gate (Stage 6 → 7)**: No PR is merged to `main` without explicit founder approval — this is a plan gate variant. Auto-approve rules in Hands-off mode do not apply.

### 10.4 Gate Enforcement Pattern

When a bot believes a workstream is not ready to advance:
1. The bot posts a plain-English gate challenge in its own channel — **not a rejection, a question**
2. The bot proposes what is missing ("Design hasn't signed off on the dashboard flow yet")
3. The founder can override by providing the missing input directly ("I'm happy with the current design, let's proceed")
4. The bot accepts the override and documents it: "Moving to Stage 5 on your direction — design not formally signed off, proceeding by exception"
5. Ops is notified of every exception override and logs it in the active workstream record

### 10.5 Ops as Stage Tracker

Ops knows the current stage of every active workstream. When a founder asks "Where are we with [feature]?", Ops answers with:
- The current stage name
- What was last completed
- What the current blocker or pending gate is
- Who needs to act next

Ops updates stage records automatically when any bot posts a gate sign-off or completion message. Stage transitions are not manually entered by the founder.

### 10.6 User Stories

**US-030**: As a founder, I want every feature to have a clear stage so I always know how far along we are.
- AC: Every active workstream has a current stage visible in #ops on request
- AC: Ops answers "Where are we with X?" with: current stage, last completed milestone, next gate, and who acts next
- AC: Stage changes are announced by Ops in #ops when they happen

**US-031**: As a founder, I want bots to challenge me if I try to skip a critical gate so I don't ship broken work.
- AC: If Engineering starts implementing without a Design sign-off, Engineering posts a gate challenge — not a refusal
- AC: The challenge proposes what is missing and offers a path forward
- AC: If the founder provides the missing input or explicitly overrides, the bot accepts it and documents the exception
- AC: Ops is notified of every exception and logs it

**US-032**: As a founder, I want design sign-off to be required before build starts regardless of my working style.
- AC: Design sign-off at Stage 3 is non-negotiable across all three working styles
- AC: In Hands-off mode, Ops prompts the founder explicitly for design approval — it is never auto-approved
- AC: If the founder is in Balanced or Hands-on mode, design sign-off appears as a plan card requiring approval before Stage 4 unlocks

**US-033**: As a founder, I want merges to always require my approval regardless of working style.
- AC: Hands-off auto-approve rules explicitly exclude PR merge actions
- AC: Every merge plan card requires explicit founder approval regardless of action cost
- AC: No cross-bot chain or scheduled trigger can auto-merge a PR

**US-034**: As a founder, I want QA to be able to waive its own sign-off if I explicitly accept the risk.
- AC: QA can propose a risk waiver with plain-English explanation of what wasn't tested and why
- AC: The founder approves or rejects the waiver — QA cannot waive unilaterally
- AC: If waived, QA's sign-off record notes: "Waived by founder — [risk summary]"

**US-035**: As a founder, I want Ops to classify the size of a feature before work starts so I know what to expect.
- AC: When a new workstream is opened, Ops (or the Product Owner) proposes a complexity tier (Hotfix / Small / Medium / Large) in plain English
- AC: The founder confirms or adjusts the tier
- AC: The confirmed tier determines which stages apply and is shown in the workstream status

**US-036**: As a founder, I want a post-ship smoke check to confirm the feature works in production before we close it out.
- AC: After a PR is merged, QA bot posts a smoke check in #qa within 5 minutes
- AC: The smoke check lists the critical paths to verify manually (plain English, no code)
- AC: Ops marks the workstream as Shipped only after the smoke check is acknowledged by the founder
