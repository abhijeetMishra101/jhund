## Findings

### [MINOR] F-001: executor.ts re-executes a plan if called a second time

- **Check**: #7 ‚Äî Plan gate bypass
- **File**: `lib/github/executor.ts:22`
- **Issue**: `executePlanActions(planId, workspaceId)` does not verify that the plan is in `status='approved'` before executing. The approve route sets status to `'approved'` and then calls the executor asynchronously via `waitUntil`. If the approve endpoint were called twice in rapid succession before the status update propagates, both calls could enter the executor and execute GitHub actions twice. The action cap RPC is atomic, so budget is protected, but the GitHub action (e.g. PR comment) would be posted twice.
- **Fix**: At the start of `executePlanActions`, atomically set `status='executing'` only if current status is `'approved'`. If the update returns 0 rows (already executing/executed/failed), return early.
- **Status**: Fixed in this PR (see code change below)

---

### [MINOR] F-002: GitHub callback skips CSRF state validation for `setup_action=update`

- **Check**: #3 ‚Äî Installation token isolation
- **File**: `app/api/github/callback/route.ts:17`
- **Issue**: When GitHub sends `setup_action=update` (triggered when a founder accepts a permission change on an existing GitHub App installation), the state cookie check is skipped. An attacker who can trick an authenticated founder into visiting a crafted URL of the form `/api/github/callback?installation_id=ATTACKER_ID&setup_action=update` could link the attacker's GitHub installation to the founder's workspace. This would give the attacker's repos access to Clan Bot's write permissions.
- **Exploitability**: Requires the victim to be logged in AND click a crafted URL while GitHub sends the `setup_action=update` parameter. Low likelihood.
- **Fix**: Add the founder's `workspace_id` to the state cookie and verify it matches after the `update` flow completes. For now, accepted as known risk ‚Äî document and revisit before first external users are added.
- **Status**: Accepted risk ‚Äî documented here. Tracked for v1.1.

### [MINOR] F-003: Installation ID logged in webhook handler

- **Check**: #1 ‚Äî Webhook signature
- **File**: `app/api/webhooks/github/route.ts:21`
- **Issue**: `installationId` is included in the `console.log` call: `'[webhook] event:', eventType, '| installationId:', installationId`. Installation IDs are not secrets (they're visible to anyone who installs the app), but logging them unnecessarily inflates log output and creates noise in security audits.
- **Fix**: Remove `installationId` from the log line, or move it to a debug-level log.
- **Status**: Fixed in this PR.
